# Presentation output: AirPlay, second display, and floating window

> Give the audience their own screen ‚Äî automatically on external displays, via AirPlay, or as a floating window ‚Äî using platform APIs for native behavior.

## Status

- **Phase:** Draft
- **Type:** feature

## Changelog

- Add audience output to external displays, AirPlay, and floating windows with automatic detection and manual override

## Motivation

The app already has `AudienceView` (full-bleed slide display) and `PresenterView` (current + next + notes), but there's no way to route the audience view to a second screen. The presenter window currently opens centered on the primary display with no screen detection. For a photography portfolio review tool, sending a clean image to the audience display is the core use case.

### Goals

1. **Zero-config when possible:** plug in a display or connect AirPlay ‚Üí audience view appears automatically
2. **Manual override always available:** user can choose which screen, or use a floating window
3. **Platform-native behavior:** use `NSScreen` on macOS, `UIScreen`/`AVExternalDisplaySession` on iOS ‚Äî no custom abstractions where the platform already handles it
4. **Graceful degradation:** if no external display, offer a floating window (macOS) or AirPlay prompt (iOS)

## Prior Art & Research

### macOS: NSScreen + NSWindow

**How other apps do it:** Keynote, PowerPoint, and Preview all use `NSScreen.screens` to detect displays and position windows. Keynote auto-detects the external display and puts slides there, presenter on the built-in screen.

**Platform APIs:**
- `NSScreen.screens` ‚Äî array of all connected screens (index 0 = primary)
- `NSApplication.shared.presentationOptions` ‚Äî hide dock/menu bar on presentation screen
- `NSWindow.setFrame(_:display:)` ‚Äî position window on a specific screen
- `NSWindow.toggleFullScreen(_:)` ‚Äî native full-screen on any screen
- `NSScreen.didChangeNotification` ‚Äî hot-plug detection (connect/disconnect during presentation)

**Key UX patterns from Keynote:**
- Auto-selects external display for audience, built-in for presenter
- Swap displays button in presenter view
- Falls back to single-screen mode if no external display
- Full-screen on audience display (not just maximized)

### macOS: Floating window

**When useful:** single-screen setups, screen sharing in video calls, side-by-side with other apps.

**Platform APIs:**
- `NSWindow.Level.floating` ‚Äî stays above normal windows
- `NSPanel` ‚Äî lightweight floating panel (HUD-style)
- `NSWindow.collectionBehavior = [.canJoinAllSpaces, .fullScreenAuxiliary]` ‚Äî follows spaces, can coexist with full-screen apps

**UX considerations:**
- Should be resizable and freely positionable
- Remember last position/size per display configuration
- Toggle between floating and full-screen

### iOS: External display + AirPlay

**Platform APIs (iOS 16+):**
- `AVExternalDisplaySession` ‚Äî manages external display lifecycle (replaces deprecated `UIScreen.didConnectNotification`)
- `UIScene` / `UIWindowScene` ‚Äî each external display gets its own scene
- Scene manifest in Info.plist with `UISceneSessionRoleExternalDisplayNonInteractive`
- SwiftUI: `WindowGroup` with scene session role for external display

**AirPlay specifics:**
- AirPlay Mirroring: system handles it, no app code needed (but mirrors the whole screen)
- AirPlay as External Display: app can provide dedicated content via `UIWindowScene`
- Control Center ‚Üí Screen Mirroring ‚Üí select Apple TV ‚Üí system creates external display scene

**Key UX patterns:**
- Keynote on iPad: detects AirPlay/HDMI and shows presenter view on iPad, slides on external
- Photos app: shows full-screen photo on external display
- No custom AirPlay picker needed ‚Äî system UI handles connection

### Cross-platform considerations

| Capability | macOS | iOS |
|-----------|-------|-----|
| Detect external display | `NSScreen.screens` | `UIApplication.shared.connectedScenes` |
| Hot-plug events | `NSScreen.didChangeNotification` | Scene delegate callbacks |
| Position on display | `NSWindow.setFrame` | Scene session role in manifest |
| Full-screen | `NSWindow.toggleFullScreen` | Automatic for external scene |
| Floating window | `NSWindow.Level.floating` | N/A (no windowing) |
| AirPlay | N/A (system mirroring) | `AVExternalDisplaySession` / scene role |
| Remember display choice | `UserDefaults` | `UserDefaults` |

## Design

### Approach

#### Display target abstraction

Introduce a `DisplayTarget` enum in SlideshowKit that describes WHERE the audience view should go:

```swift
enum DisplayTarget: Equatable, Sendable {
    case automatic           // pick the best available option
    case screen(screenID: String)  // specific display by persistent ID
    case floatingWindow      // macOS only: resizable floating panel
    case none                // no audience output
}
```

The **app layer** (not SlideshowKit) maps this to platform-specific window management. SlideshowKit stays platform-independent.

#### macOS implementation

1. **Screen detection service** ‚Äî wraps `NSScreen.screens` and `didChangeNotification`
2. **Audience window controller** ‚Äî manages the NSWindow for audience output
   - Positions on selected screen, goes full-screen
   - Hides cursor, dock, menu bar on audience screen
   - Hot-plug: if audience screen disconnects, gracefully falls back to floating or none
3. **Floating window mode** ‚Äî `NSPanel` at `.floating` level, resizable, remembers frame
4. **Display picker UI** ‚Äî dropdown/menu in presenter toolbar showing available screens + "Floating Window" + "None"

**Auto-detection logic:**
1. If 2+ screens: audience ‚Üí external (non-primary), presenter ‚Üí primary
2. If 1 screen: offer floating window or no audience output
3. User override persists until display configuration changes

#### iOS implementation

1. **External display scene** ‚Äî declare `UISceneSessionRoleExternalDisplayNonInteractive` in Info.plist
2. **Scene delegate** ‚Äî when external scene connects, show `AudienceView` in that scene's window
3. **Presenter on device** ‚Äî iPad/iPhone always shows `PresenterView` when an external display is active
4. **AirPlay** ‚Äî handled by the same scene mechanism (no separate code path)

#### Shared state

Both presenter and audience observe the same `Slideshow` model (already works this way). No new state synchronization needed ‚Äî `@Observable` handles it.
