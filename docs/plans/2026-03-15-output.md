# Presentation output: display targeting and output window

> Move the output window to any connected display via context menu or keyboard — manual control, no auto-detection magic.

## Status

- **Phase:** Draft
- **Type:** feature

## Changelog

- Output window with display targeting: right-click to move between screens, F for full-screen, D to cycle displays
- Frame/display persistence across sessions

## Motivation

The app has a presentation output window (Play button, Cmd+Shift+P) that shows a full-bleed image + caption on black. But it always opens centered on the primary screen with no way to target an external display. For photography portfolio reviews — where the photographer sits with their laptop and the audience looks at a projector or external monitor — sending the clean image to a specific screen is the core use case.

### Goals

1. **User controls placement:** right-click context menu to pick a display, keyboard shortcut to cycle
2. **No auto-detection magic:** plug in a display, then explicitly tell the app where to present — no surprises
3. **Platform-native behavior:** `NSScreen` enumeration, native full-screen via `toggleFullScreen`, standard window management
4. **Minimal new code:** ~100-150 lines, no new types, all in the app target

### Non-goals (future work)

- iOS / AirPlay / external display via scene roles — deferred until iOS target exists
- Auto-detection (Keynote-style automatic routing to external display)
- Presenter controls view (current + next + notes on primary screen)
- Auto-hiding toolbar overlay (QuickTime-style)
- Picture-in-picture / always-on-top floating mode
- Display color profile management

## Prior Art & Research

### Approaches considered

**A: Output = a second window (no display awareness).** User manually drags the window. Dead simple but clunky for multi-screen. This is what we have today.

**B: Smart display routing (Keynote-style).** Auto-detect external displays, route audience view there, keep presenter controls on primary. ~400-500 lines, needs ScreenDetector, hot-plug handler, display persistence, clamshell detection, and a rebuilt presenter controls view. Large edge case surface (3+ screens, mirroring, Sidecar, Stage Manager).

**C: Output window + "Move to Display" menu (chosen).** Start from A, add a context menu with screen enumeration and a keyboard shortcut to cycle displays. ~100-150 lines, no new types. User stays in control. Clean upgrade path to B later — add ScreenDetector and auto-routing on top, keep the menu as manual override.

### Why C over B

The primary use case is a photographer setting up deliberately for a review session — not a conference speaker who needs instant auto-detection. "Plug in, press one menu item" is good enough. The 3-4x complexity of B isn't justified yet. If auto-detection proves necessary, C's manual menu becomes B's override UI, so no work is thrown away.

### Platform APIs used

- `NSScreen.screens` — array of all connected screens (index 0 = primary)
- `NSScreen.localizedName` — human-readable display name (macOS 10.15+)
- `NSWindow.setFrame(_:display:)` — position window on a specific screen
- `NSWindow.toggleFullScreen(_:)` — native macOS full-screen (creates a new Space)
- `NSWindow.screen` — which screen the window is currently on
- `NSCursor.setHiddenUntilMouseMoves(_:)` — auto-hide cursor during presentation
- `CGDisplayCreateUUIDFromDisplayID` — persistent display UUID (survives reboots, unlike `CGDirectDisplayID`)
- `UserDefaults` — persist frame/display choice

## Design

### Output window (already implemented)

`PresenterView` — full-bleed image + caption overlay on black. Keyboard nav: arrows, space, escape. Hosted in an NSWindow created by `openPresenterWindow()` in `SlideshowDocumentView`. Window has transparent titlebar (close/zoom buttons appear on hover), hidden title, black background.

Both the output window and the main editor observe the same `@Observable Slideshow` model. Navigating slides in either window updates both. No new synchronization needed.

### Context menu

Right-click anywhere on the output window shows:

```
Move to Display ▸
  ✓ Built-in Retina Display
    DELL U2723QE
Full Screen                    F
─────────────────────────────────
5 / 24
```

- **Move to Display** — submenu with one item per `NSScreen.screens`, checkmark on current screen
- **Full Screen** — toggles native full-screen, shows current state
- **Slide counter** — informational label (not a button), shows `selectedIndex + 1 / slides.count`

Implementation: SwiftUI `.contextMenu { }` modifier on `PresenterView`. Screen enumeration reads `NSScreen.screens` at menu build time (always fresh).

**Window reference:** inject the `NSWindow` instance into `PresenterView` via a custom `EnvironmentKey` (set by `openPresenterWindow()` after window creation). Do **not** use `NSApp.keyWindow` — it's brittle in a multi-window app (the output window may not be key when a context menu action fires if focus shifted to the editor). The injected reference guarantees we always target the correct window.

When only 1 screen is connected, the "Move to Display" submenu shows one item with a checkmark. No special-casing needed — the menu is just less useful.

### Keyboard shortcuts

Added to PresenterView via `.onKeyPress`:

| Key | Action | Notes |
|-----|--------|-------|
| ← | Previous slide | Existing |
| → | Next slide | Existing |
| Space | Next slide | Existing |
| Escape | Close output window | Existing |
| **F** | **Toggle full-screen** | Via injected window reference |
| **D** | **Cycle to next display** | See below |

**D key behavior (windowed mode):**
1. Get current screen from injected window reference
2. Find index in `NSScreen.screens`
3. Move to `screens[(index + 1) % screens.count]`
4. `window.setFrame(targetScreen.visibleFrame, display: true)`

**D key while full-screen:** `toggleFullScreen` is asynchronous — calling `setFrame` immediately after will fail or cause visual glitches. Implementation must:
1. Call `window.toggleFullScreen(nil)` to exit full-screen
2. Wait for `windowDidExitFullScreen(_:)` delegate callback (requires `NSWindowDelegate` on the window)
3. Only then call `setFrame` to move to the next screen
4. Optionally re-enter full-screen on the new screen via another `toggleFullScreen` call

This requires the window to have an `NSWindowDelegate` — set it up in `openPresenterWindow()`. The delegate can be a lightweight class (or the existing close-notification handler can be replaced with delegate methods).

### Cursor auto-hide

On each slide transition, call `NSCursor.setHiddenUntilMouseMoves(true)` on `@MainActor` (this is an AppKit UI call — must be on main thread). The cursor reappears when the user moves the mouse (e.g., to right-click the context menu) and hides again on the next slide change. This is the standard macOS pattern used by QuickTime Player and Preview in slideshow mode. Since `PresenterView` is in a SwiftUI view body and slide transitions happen via `@Observable` model updates on `@MainActor`, the call site is already main-actor-isolated.

### Frame persistence (branch 2)

**Save:** when the output window closes, store:
- Display UUID via `CGDisplayCreateUUIDFromDisplayID` (persistent across reboots, unlike `CGDirectDisplayID` which changes for Thunderbolt docks and some display types)
- `window.frame` as `NSRect`
- Keyed by slideshow folder URL in `UserDefaults`

**Restore:** when Play is pressed:
1. Look up saved display UUID and frame for the current slideshow
2. Enumerate `NSScreen.screens`, match by converting each screen's `NSScreenNumber` → `CGDisplayCreateUUIDFromDisplayID`
3. If saved display is still connected: restore frame on that screen
4. Otherwise: center on primary screen (current behavior)

### Where code lives

All changes in the **Slideshow app target**:

| File | Changes |
|------|---------|
| `Slideshow/Views/PresenterView.swift` | Add `.contextMenu`, `.onKeyPress("f")`, `.onKeyPress("d")`, cursor hide, `NSWindow` environment key |
| `Slideshow/SlideshowApp.swift` | Window environment injection, `NSWindowDelegate` for full-screen transitions, frame save/restore |

**No SlideshowKit changes.** One new lightweight type: an `NSWindowDelegate` class for handling `windowDidExitFullScreen` (replaces the current `NotificationCenter` observer for window close). Display management is platform-specific AppKit code that belongs in the app target.

**Delete:** `Slideshow/Views/AudienceView.swift` — duplicates PresenterView, unused.

### Edge cases

| Scenario | Behavior |
|----------|----------|
| 1 screen | Menu shows one item with checkmark. D key is a no-op. |
| Mirroring mode | `NSScreen.screens` returns 1 screen. Same as single screen. |
| Display disconnect during presentation | macOS moves window to another screen automatically. Menu updates on next right-click. |
| 3+ screens | Menu shows all screens. D key cycles through all. |
| Sidecar iPad | Appears in `NSScreen.screens` with its name. Works like any external display. |
| Stage Manager | No impact — Stage Manager manages normal windows. Full-screen exits Stage Manager as usual. |
| Clamshell mode (lid closed) | 1 screen in `NSScreen.screens`. Same as single screen. |
| Play pressed with no slides | Button is `.disabled(slideshow.slides.isEmpty)`. Can't happen. |
| Output window open, different slideshow opened | Shared `Slideshow` model updates. Output window shows new slideshow. |

### Window chrome decision

**Chosen: minimal titlebar (Approach B from brainstorming).** Transparent titlebar with close/zoom buttons on hover. Standard macOS window behavior, familiar, already implemented.

**Future option: auto-hiding toolbar overlay.** QuickTime-style controls that appear on mouse move and disappear on idle. Not in scope — adds animation/timer complexity for minimal UX benefit when the context menu covers all actions.
