# Support plain folders with optional slideshow.yml project file

> Drop the `.slideshow` bundle requirement. Any folder of images is a valid project. An optional `slideshow.yml` project file adds title and future layout metadata.

## Motivation

The app currently requires `.slideshow` bundles (folders with `com.apple.package` UTType). This adds friction: users can't point at an existing photo folder and present. "Mappe" (German for folder/portfolio) reinforces the core insight — a slideshow project IS a folder.

The `.slideshow` bundle adds no real value internally. FolderScanner, FileReorderer, BookmarkManager, and all models already treat it as a plain directory. The bundle is cosmetic (Finder icon) and limiting (can't open regular folders).

## Design

### Project file format

`slideshow.yml` — visible, YAML, human-editable. Serves as a project marker in Finder.

```yaml
version: 1
title: Paintings That Tell Secrets
```

- Both fields optional. Missing `version` defaults to 1. Missing `title` → folder name.
- Unknown keys preserved on round-trip (same pattern as sidecar frontmatter).
- `003--` prefix ordering unchanged — project file does NOT replace filename-based ordering.

### What changes

| Area | Before | After |
|------|--------|-------|
| Open flow | `.folder` + `UTType("is.kte.slideshow")` | `.folder` only |
| New project | NSSavePanel creates `.slideshow` bundle | NSOpenPanel creates/selects a folder, writes `slideshow.yml` |
| Display name | Strip `.slideshow` extension | Project file title, fallback to folder name |
| UTType | `is.kte.slideshow` conforming to `com.apple.package` | Removed entirely |
| Info.plist | `UTExportedTypeDeclarations` block | Removed (plist may become empty) |
| FolderScanner | Returns `[Slide]` | Returns `ScanResult` (slides + optional project file) |
| Slideshow model | No project file awareness | `projectFile: ProjectFile?` property |

### What does NOT change

- `FileReorderer` — works on any folder
- `SidecarParser` / `SidecarWriter` — format unchanged
- `Slide` model — no changes
- `BookmarkManager` — already folder-agnostic
- `ImageCache`, `EXIFReader` — unrelated
- `003--` prefix ordering — stays

### Approach (implementation order)

1. **ProjectFile model + parser + writer** in SlideshowKit (TDD, follows SidecarParser/Writer pattern)
2. **FolderScanner integration** — new `ScanResult` return type, `slideshow.yml` skipped from image/sidecar classification
3. **Slideshow model** — `projectFile` property, `name` derived from project file title
4. **App layer** — simplify fileImporter, remove UTType, update openSlideshow/createNew flows
5. **Examples** — rename `.slideshow/` → plain folders, add `slideshow.yml` to each
6. **Documentation** — CLAUDE.md, rules, README

### Future: multi-image layouts (not in scope)

The project file will later support combining ordered images onto one presentation slide:

```yaml
version: 1
title: My Portfolio
layouts:
  - images: [003--sunset.jpg, 004--sunrise.jpg]
    axis: horizontal  # default; or vertical
  - images: [007--a.jpg, 008--b.jpg, 009--c.jpg, 010--d.jpg]
    arrangement: grid
```

Images not in `layouts` get their own slide. Order still comes from filenames.
